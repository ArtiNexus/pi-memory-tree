# MEMORY*.md 模板规范

> 本文件定义了层级记忆树所有枝条文件的格式规范。
> 新建枝条文件时，以此为模板。
> 版本: v1 | 生效: 2026-05-30

---

## 1. 命名规则

### 格式

```
MEMORY{N}_{type}.md
```

| 部分 | 规则 | 示例 |
|:-----|:-----|:------|
| `N` | 正整数，按创建顺序递增 | MEMORY1, MEMORY2, MEMORY16 |
| `type` | 记忆类型缩写（见下方） | sem, epi, pro, mix |
| 子枝杈 | `MEMORY{N}-{M}_{type}.md` | MEMORY1-1_sem.md, MEMORY2-3_epi.md |
| 孙枝杈 | `MEMORY{N}-{M}-{K}_{type}.md` | MEMORY1-1-1_sem.md |

### 记忆类型

| 类型 | 缩写 | 含义 | 适合内容 | 检索策略 |
|:-----|:----:|:-----|:---------|:---------|
| **语义型** | `sem` | 事实性知识，静态可查 | 环境配置、工具版本、用户偏好 | 关键词精准定位，读完后直接回答 |
| **情节型** | `epi` | 发生过的事，时间线重要 | 升级记录、项目历程、事件经过 | 按时间顺序读取，关注先因后果 |
| **程序型** | `pro` | 怎么做，步骤重要 | 工作流、操作手册、配置步骤 | 按步骤读取，关注顺序和条件分支 |
| **混合型** | `mix` | 兼有多种 | Daemon 综合、主题综述 | 按需混合检索，标记出各段落的类型 |

### 根文件（不遵循此命名）

| 文件 | 层级 | 用途 |
|:-----|:----:|:------|
| `MEMORY.md` | Level 0 | INDEX 索引（关键词→文件映射），每次会话必加载 |
| `USER.md` | Level 0 | 用户画像 |
| `WATCH.md` | Layer F | 当前焦点看板 |
| `MEMORY_TEMPLATE.md` | — | 本模板 |

## 3. 各区块规范

### 3a. 文件头（第 1-4 行）

```
# MEMORY1_sem: 环境配置
                    ↑ 严格等于文件名（不含 .md）      ↑ 简明确切的中文描述

> 最后更新: YYYY-MM-DD         ← 此文件最后一次被修改的日期
> 创建于:   YYYY-MM-DD         ← 此文件首次创建的日期
```

### 3b. 关联枝

**何时必须有**：只要这个枝与其他枝存在主题交集。

```
═══ 关联枝 ═══
| 关联枝条 | 关联原因 | 关键交集 |
|:---------|:---------|:---------|
| MEMORY4_mix | Daemon 依赖本机环境 | 系统版本/Proxy |
| MEMORY2-1_sem | 飞书通道需要 Proxy | Proxy 配置 |

```

**关联枝条列**：指向的 `MEMORY{N}_{type}.md` 文件名（不含路径）。

**关联原因**：一句话说清为什么两者有关。常见原因：

| 原因类型 | 例句 |
|:---------|:-----|
| 依赖关系 | "Daemon 依赖本机环境" |
| 共享主题 | "都涉及网络代理配置" |
| 因果链条 | "安装 X 导致需要 Y" |
| 前提条件 | "使用此工具需先配置 Proxy" |

**关键交集**：具体哪几个子话题是两枝共有的，限定关联范围。

**双向关联规则**：A 关联 B 时，B 也应关联 A。但不在**创建时**强制要求双向填充，而在**下次分叉或修改时**检查补齐。

```
创建 MEMORY1_sem → 关联 MEMORY4_mix ✅
                   ↓
下次修改 MEMORY4_mix → 发现缺失反向关联 → 自动补上 ✅
```

### 3c. 子索引

**何时有**：当此文件有子枝杈文件时。

```
═══ 子索引 ═══
| 子主题 | 文件 | 简介 |
|:-------|:-----|:------|
| 开发工具 | MEMORY1-1_sem | 各语言/工具版本详情 |
| Proxy配置 | MEMORY1-2_sem | FlClash 网络代理设置 |
```

**分叉命名规则**：父文件为 `MEMORY{N}_{type}.md`，第一个子文件为 `MEMORY{N}-1_{type}.md`，第二个为 `MEMORY{N}-2_{type}.md`，以此类推。

**深层命名**：`MEMORY{N}-{M}-{K}_{type}.md`，不限深度。

### 3d. 内容条目

#### 分隔符

每个条目以 `§` 开头，后接一个空格。

```
§ [2026-05-29]
内容...
```

#### 时间戳格式

- **首次记录**：`§ [YYYY-MM-DD]`
- **更新**：`§ [YYYY-MM-DD] ⬆ 更新`
- **废弃**：`§ [YYYY-MM-DD] ⛔ 废弃`（条目保留不删，仅标记）
- **无日期条目**：只有确实不需要时间追踪的条目（如说明性文字）可用 `§` 无日期

#### 内容风格

- **紧凑**：能用一行说清的就用一行
- **结构化**：多属性用 `|` 分隔或用列表
- **不存原始对话**：只存提炼后的事实

```
✅ 好的：
§ [2026-05-29]
macOS 15.7.3 | CPU:x86_64/8C | RAM:16G | SSD:233G/42Gfree

❌ 不好的：
§ [2026-05-29]
2026年5月29日，用户告诉我他的电脑是macOS 15.7.3，处理器是x86_64，有8个核心，内存16GB，硬盘233GB里面还有42GB的空闲空间。
```

### 3e. 历史版本

**何时有**：当文件中某个条目被更新时，旧版本移到此处。

```
═══ 历史版本 ═══

2026-06-15:
  Java: OpenJDK 26.0.1 → 27.0.0
  Go: 1.26.3 → 1.27.0

2026-07-01:
  RAM: 16GB → 32GB
```

**何时不保留**：
- 第一次记录
- 明显输入错误（直接删除错误条目即可）
- 内容没有本质变化（仅修正拼写/格式）

## 4. 分叉规则

### 触发条件

文件内容（`═══ 内容 ═══` 区块）达到 **2000 字符**时触发分叉。

> 留 200 字符给文件头 + 关联枝 + 子索引等结构化区块。

### 分叉步骤

```
MEMORY1_sem.md 内容达到 2000 chars
  ↓
① 分析内容，找出 >2 条可独立成枝的主题
  例：[环境配置] 和 [开发工具] 和 [Proxy配置]
  ↓
② 父文件保留：最核心的概览 + 新增子索引
  ↓
③ 新建 MEMORY1-1_sem.md ← 子主题 1 详情
  ↓
④ 新建 MEMORY1-2_sem.md ← 子主题 2 详情
  ↓
⑤ 父文件加关联枝（如果子文件之间有关联）
  ↓
⑥ INDEX.md 更新关键词路径（如适用）
```

### 分叉后的父文件例子

```markdown
# MEMORY1_sem: 环境配置

> 最后更新: 2026-07-01
> 创建于: 2026-05-29

═══ 关联枝 ═══
| 关联枝条 | 关联原因 | 关键交集 |
|:---------|:---------|:---------|
| MEMORY4_mix | Daemon 环境依赖 | 系统版本/Proxy |

═══ 子索引 ═══
| 子主题 | 文件 | 简介 |
|:-------|:-----|:------|
| 开发工具 | MEMORY1-1_sem | Node/Python/Go/Java/Rust 各版本 |
| Proxy配置 | MEMORY1-2_sem | FlClash 代理设置详情 |

═══ 内容 ═══

§ [2026-05-29]
macOS 15.7.3 (Sequoia), x86_64, 8-core, 16GB RAM, 233GB SSD (42GB free)

§ [2026-07-01] ⬆ 更新
RAM: 16GB → 32GB（升级详情见 MEMORY1-3_epi）
```
